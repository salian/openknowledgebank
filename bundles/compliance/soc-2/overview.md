# Overview

SOC 2 is part of AICPA/CIMA's System and Organization Controls suite. This bundle treats AICPA/CIMA materials as the official source category, auditor requests as engagement-specific evidence, and user-provided system records as necessary for organization-specific conclusions.

## What This Bundle Is For

- Triage whether a SOC 2 question is about report scope, trust services categories, Type I versus Type II readiness, system description, evidence collection, customer response, or professional-review handoff.
- Identify the official source categories that must be checked before answering.
- Produce source-aware readiness briefs that separate verified facts from assumptions and missing evidence.
- Prevent unsupported claims that an organization is SOC 2 compliant, audit-ready, or covered for a specific customer requirement.

## What This Bundle Is Not For

- It is not legal, accounting, CPA, audit, attestation, or control-design advice.
- It is not a copy of AICPA criteria, points of focus, guidance, report templates, or opinion language.
- It does not decide control sufficiency, evidence sufficiency, operating effectiveness, or audit outcomes.
- It does not replace the service auditor, legal counsel, security owner, or compliance owner.

## Source Discipline

Every SOC 2 output should distinguish:

- verified official-source facts;
- the AICPA/CIMA source category checked for the task;
- facts provided by the user;
- assumptions made only for a draft;
- missing facts, source material, or professional review that block a conclusion.

## Risk Boundary

Contacting auditors, sending SOC 2 reports or bridge letters, answering customer security questionnaires, exporting evidence, changing controls, changing policies, or modifying production systems are risky actions. An agent should require explicit user confirmation and qualified professional review before those actions.
