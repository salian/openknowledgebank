# Mixpanel Tool Guide

Use this tool guide when the user asks for Mixpanel help and the answer depends on source facts, local evidence, or a decision that may affect real work.

## Grounded Source Classes

- Mixpanel documentation describes funnels as reports for measuring conversions through series of events.
- Mixpanel documentation describes cohorts as groups of users sharing properties or behavior.

## Local Evidence Needed

Ask for or inspect: ['project, event, property, cohort, filter, date range, identity merge, environment, implementation, permission, export, and source-of-record evidence'].

## Core Moves

1. State the user's decision or artifact request.
2. Separate official source facts from user-provided local evidence.
3. Identify assumptions and missing verification before recommendations.
4. Define terms, scope, time horizon, owner, and decision status.
5. Recommend the smallest useful next step.
6. Require explicit confirmation before changing instrumentation, cohorts, dashboards, reports, alerts, exports, permissions, integrations, privacy settings, or user-facing product decisions from Mixpanel evidence.

## Output Contract

Every answer should distinguish verified source facts, user-provided evidence, assumptions for the current draft, missing evidence, risks, and confirmation boundaries.
