# Salesforce Service Cloud Source Checks

## Official Source Categories

- Salesforce product pages for high-level product positioning and naming.
- Salesforce Help for setup, feature behavior, release, and admin guidance.
- Salesforce Developers documentation for object references, REST API, Metadata API, Tooling API, and platform developer behavior.
- Salesforce release notes for current feature naming, availability, and changed behavior.
- Salesforce pricing, packaging, contract, and org license evidence when edition, entitlement, add-on, or commercial availability matters.

## Local Org Evidence

Ask the user for the relevant local evidence before final claims:

- org, release, edition, license, package, and enabled feature evidence;
- object, field, record type, page layout, validation rule, permission, and sharing evidence;
- queue, skill, presence status, routing, assignment rule, escalation rule, flow, entitlement, milestone, macro, quick action, and channel evidence;
- report type, report filters, dashboard filters, ownership/sharing, time zone, formula, and refresh evidence;
- integration user, API version, auth scope, endpoint, payload, limit, and audit evidence;
- AI grounding source, prompt/action configuration, handoff, test, escalation, and human approval evidence.

## Claims That Need Verification

Do not make authoritative claims about these without current source or local evidence:

- custom object, field, record type, picklist, queue, skill, or routing names;
- report numbers, SLA metrics, CSAT definitions, backlog definitions, first response, resolution, escalation, or entitlement formulas;
- edition availability, license requirements, pricing, add-ons, AI feature availability, or release-specific behavior;
- API endpoint syntax, payloads, scopes, limits, or integration behavior;
- whether an automation, flow, AI agent, reply, handoff, or customer-facing action is safe to enable.
