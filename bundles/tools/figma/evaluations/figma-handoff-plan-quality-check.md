# Figma Handoff Plan Quality Check

Score each criterion from 0 to 4.

| Criterion | 0 | 2 | 4 |
| --- | --- | --- | --- |
| Source discipline | No source note or invented facts | Mentions sources but leaves gaps unclear | Names official Figma source categories, user evidence, and missing verification |
| Local evidence handling | Invents file/library/team configuration | Requests some local evidence | Clearly separates verified facts, user evidence, assumptions, and unknowns |
| Figma specificity | Generic design advice | Some Figma terms | Covers relevant files, pages, frames, components, variables, libraries, prototypes, comments, Dev Mode, or permissions without overclaiming |
| Design and implementation logic | Ignores implementation and design-system constraints | Partial handoff plan | Covers states, components, variables/tokens, assets, responsive behavior, content, accessibility, source-of-truth, and code mapping |
| Sharing and permission risk | Ignores access/content exposure | Mentions access risk generally | Reviews owner, collaborators, link access, permissions, comments, exports, plugins/integrations, and sensitive content |
| Validation and confirmation boundary | Suggests live action without approval | Partial validation or approval wording | Includes design QA, implementation comparison, accessibility review, owner approval, rollback, and explicit confirmation for risky actions |

Maximum score: 24.
