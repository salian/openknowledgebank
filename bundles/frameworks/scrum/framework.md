# Scrum Framework

Scrum is useful when a team needs a transparent way to inspect progress, adapt plans, and create increments of product value. It becomes unsafe when an agent treats ceremonies, Jira boards, velocity charts, or job titles as proof that Scrum is working.

## Core Moves

1. State the Scrum question: adoption review, sprint operation, role/accountability confusion, backlog quality, Definition of Done, stakeholder review, or improvement planning.
2. Inspect local evidence: team accountabilities, current events, artifacts, commitments, workflow, constraints, product context, and metrics.
3. Separate official Scrum framework terms from local process choices.
4. Mark each issue as source-grounded, user-provided, assumed for the draft, or needing verification.
5. Recommend the smallest inspectable next step before broad process changes.
6. Require explicit confirmation before changing live tools, sprint commitments, staffing responsibilities, release plans, or stakeholder communications.

## Interpretation Rules

- Scrum is not the same thing as Agile, Jira, sprints, story points, standups, or velocity.
- A team can run events and still lack transparency, inspection, adaptation, product focus, or a usable Increment.
- Local adaptations need context: classify them before calling them harmful, necessary, or outside Scrum.
- The Product Owner, Scrum Master, and Developers are accountabilities, not permission to invent an organization's job architecture.
- The Definition of Done, Sprint Goal, and Product Goal should be treated as commitments that need local evidence.

## Output Contract

Every Scrum recommendation should distinguish:

- verified framework facts from the Scrum Guide;
- user-provided evidence about the current team or organization;
- assumptions for the current draft;
- missing verification before conclusion or action.
