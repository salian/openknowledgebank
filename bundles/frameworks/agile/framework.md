# Agile Framework

Agile is an umbrella framework for adaptive software delivery. It is most useful when guidance keeps customer value, frequent usable delivery, collaboration, responsiveness to change, technical quality, sustainable pace, simplicity, empowered teams, and regular improvement visible.

## Core Moves

1. State the delivery decision or process problem.
2. Inspect the user's current evidence: workflow, roles, feedback loops, release cadence, delivery metrics, quality signals, constraints, and customer collaboration.
3. Separate Agile umbrella principles from method-specific mechanics.
4. Identify which principle themes are strong, weak, unknown, or blocked by constraints.
5. Choose a method family only after matching context and tradeoffs.
6. Recommend small, inspectable changes before broad reorganizations.
7. Require explicit confirmation for live process, staffing, contract, tool, reporting, or customer-facing changes.

## Interpretation Rules

- Agile is not synonymous with Scrum, sprints, velocity, Jira, or standups.
- A team can perform ceremonies and still lack meaningful feedback, working software, technical excellence, or adaptation.
- A team can use Agile principles without adopting every named method artifact.
- Method choice should respond to work type, uncertainty, flow, risk, dependencies, scale, and governance.
- Local evidence matters: do not judge maturity or prescribe changes from labels alone.

## Output Contract

Every Agile recommendation should distinguish:

- verified source facts from the Manifesto or other named sources;
- user-provided evidence about the current team or organization;
- assumptions for the current draft;
- missing verification before conclusion or action.
