# Map Technical Dependencies

Use this workflow when a roadmap item or feature requires coordination across teams, services, data, APIs, vendors, or policies.

## Inputs

- Initiative goal and target launch window if known.
- Known teams, systems, services, APIs, data flows, repos, vendors, policies, or customer commitments.
- Current status from tickets, plans, specs, architecture notes, or owner updates.
- Risk tolerance and decision owner.

## Steps

1. Identify the outcome and launch decision being supported.
2. Build a dependency table with owner, source evidence, current state, blocker, risk, decision needed, and due date if user-provided.
3. Mark each dependency as verified, user-provided, assumed, or needs verification.
4. Separate sequencing dependencies from approval dependencies and operational readiness dependencies.
5. Highlight critical path, unresolved risks, and decisions that need owner confirmation.

## Output Contract

- Direct summary of critical path and highest-risk blockers.
- Source note.
- Dependency table.
- Assumptions and missing verification.
- Recommended next checks and owner questions.
- Confirmation request before creating or changing tickets, docs, repos, flags, or communications.
