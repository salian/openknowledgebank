# Design Review

Use this workflow to critique an existing design artifact.

## Inspect First

Before giving final recommendations, establish what evidence is available:

- Artifact under review: screenshot, flow, prototype, file description, or spec.
- Product goal and user scenario.
- Design-system or brand rules.
- Accessibility target and legal/professional review needs.
- Engineering constraints and known implementation limits.

## Output Contract

The output should include:

- `Source note`: design artifact, source-of-record product/design-system evidence, accessibility reference category, and missing verification.
- Direct answer: what should change first and why.
- Severity-ranked issues, grouped by user impact.
- Flow/state coverage: happy path, empty, loading, error, permissions, edge, responsive, and assistive-technology-sensitive states.
- Evidence distinction: verified from source, provided by user, assumed for draft, and needs verification.
- Handoff notes and owner-specific questions.

## Confirmation Boundary

Recommendations may be drafted without confirmation. Changes to live design files, tickets, documentation, research repositories, or stakeholder communications require explicit user approval.
